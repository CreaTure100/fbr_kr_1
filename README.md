
## 📋 Краткое описание практик

### Практическое занятие 19: Работа с реляционными СУБД (PostgreSQL)
В данной работе был изучен процесс взаимодействия с реляционными базами данных на примере **PostgreSQL**.
* **Основные задачи:** Настройка подключения к БД через библиотеку `pg` и ORM **Sequelize**, описание моделей данных и связей между таблицами.
* **Результат:** Реализовано API для управления пользователями (CRUD) с использованием SQL-запросов и транзакций для обеспечения атомарности операций.

### Практическое занятие 20: Работа с NoSQL СУБД (MongoDB)
Практика посвящена изучению документоориентированных систем на примере **MongoDB**.
* **Основные задачи:** Установка и настройка безопасности (аутентификации) в MongoDB, использование ODM **Mongoose** для определения схем данных и взаимодействия с коллекциями.
* **Результат:** Создано API для работы с пользователями, реализующее гибкую NoSQL-структуру хранения данных без жестко заданной схемы.

### Практическое занятие 21: Кэширование с использованием Redis
Целью работы было ускорение работы приложения и снижение нагрузки на основную базу данных с помощью **Redis** — быстрого in-memory хранилища типа key-value.
* **Основные задачи:** Реализация `cacheMiddleware` для промежуточного хранения результатов GET-запросов, настройка времени жизни кэша (TTL) и механизмов его инвалидации (очистки) при обновлении данных.
* **Результат:** Оптимизированы маршруты получения списков пользователей и товаров с поддержкой актуальности данных в кэше.

### Практическое занятие 22: Балансировка нагрузки в веб-приложениях
В этой практике рассматривались механизмы распределения входящего трафика между несколькими серверами для повышения отказоустойчивости системы.
* **Основные задачи:** Настройка программного балансировщика **Nginx** (алгоритм Round Robin) и изучение альтернативы в виде **HAProxy**. Настройка проверок доступности серверов (health checks) через параметры `max_fails` и `fail_timeout`.
* **Результат:** Создана архитектура с несколькими бэкенд-серверами, способная корректно перераспределять запросы при выходе из строя одного из узлов.

### Практическое занятие 23: Контейнеризация приложений с Docker
Заключительная работа объединяет предыдущие наработки в единую изолированную среду с помощью технологий контейнеризации.
* **Основные задачи:** Написание инструкций для сборки образов в **Dockerfile**, использование **Docker Compose** для управления многоконтейнерным стеком (API Gateway, сервисы пользователей и заказов, база данных, кэш и балансировщик).
* **Результат:** Реализовано микросервисное приложение, развернутое в среде WSL, где все компоненты общаются внутри изолированной сети Docker по именам сервисов.

## 🚀 Запуск проекта (на примере микросервисов из Практики 23)

Для автоматического развертывания системы балансировки и изолированных серверов убедитесь, что у вас установлен и запущен **Docker Desktop** (или Docker Engine в Linux/WSL).

**1. Клонирование репозитория и переход в папку проекта:**
```bash
git clone https://github.com/CreaTure100/fbr_kr_1.git
cd fbr_kr_1/"practice 23"
```

**2. Сборка образов и запуск контейнеров:**
```bash
docker compose up --build
```
*(Для запуска в фоновом режиме добавьте флаг `-d`: `docker compose up -d --build`)*

**3. Проверка работы (тестирование балансировщика):**
Откройте браузер и перейдите по адресу `http://localhost/` или выполните команду в новом окне терминала:
```bash
curl http://localhost/
```
*При повторных запросах ответы будут приходить от разных экземпляров backend-сервера (сработает Round Robin).*

**4. Остановка системы:**
```bash
docker compose down
```
